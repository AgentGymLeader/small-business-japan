# Ticket Deflector JP

## Quick start

顧客のメールを転送するか貼り付けると、Stripe で注文状況を確認し、
オーナーの文体に合わせた返信下書きを作成する。
返金が必要な場合は詳細を提示してからオーナーの承認を待つ。

```
オーナー: 「この顧客に返信して」[メールを転送]
→ メールから顧客情報・課題を抽出
→ Stripe で取引状況を確認
→ 顧客履歴（kintone・スプレッドシート）を確認
→ オーナーの文体に合わせた返信下書き作成
→ オーナーが承認 → 送信またはドラフト保存
→ 返金が必要な場合: 確認プロンプト → オーナー承認 → 実行
```

## Workflow

1. **顧客メッセージを読む。** 転送された Gmail スレッドまたは貼り付けられたテキストを受け取る。以下を抽出する：顧客のメールアドレス、氏名、注文・取引 ID（あれば）、課題の核心——返金依頼、注文状況確認、または一般的なクレーム。複数の課題がある場合は記載順に対応する。

2. **Stripe で注文状況を確認する。** 顧客のメールアドレスまたは取引 ID で Stripe の取引を検索する。金額、日付、ステータス、既に返金が実施されているかを確認する。Stripe が接続されていない場合は返信内にその旨を記載して続ける。取引が見つからない場合はフラグを立てる——推測はしない。
   - **レート制限への対応:** 顧客が取引 ID を提供している場合はそれを使用する——単一レコードの検索でレート制限を避けられる。メールアドレスで検索する場合は7日間のウィンドウを使用する。
   - 複数の取引が一致する場合は全件提示し、どれが該当するかオーナーに確認する。

3. **顧客履歴を確認する。** kintone またはスプレッドシート（顧客管理）でメールアドレスを検索する。顧客ランク、過去の取引、過去のクレーム履歴を確認する。顧客情報がない場合はその旨を記載し、返信後に登録を提案する——返信フロー中には登録しない。

4. **返信下書きを作成する。** オーナーの文体で書く。課題の種類に応じてトーンを調整する：
   - 返金依頼 → 共感的・明確・行動指向
   - 注文状況確認 → 事実に基づき・安心感を与える
   - 一般的なクレーム → 状況を認め・説明し・解決策を提示

   データの不足は下書き内に角括弧で記載する（例: *[注意: Stripe で取引が見つかりません——注文番号を確認してから送信してください]*）。

5. **Approval gate — オーナーが下書きを確認する。** 完全な下書きを提示する。オーナーが承認するまで送信も保存もしない。自由に編集してから承認できる。

6. **Approval gate — 返金の実行。** 返金が必要な場合は下書き承認後に専用の確認プロンプトを表示する：

   > *「{顧客名}（{メールアドレス}）への返金 {金額}円（取引 ID: {ID}）を実行しますか？「はい」で実行します。」*

   明確な「はい」以外の返答があった場合は停止して確認する。

7. **送信またはドラフト保存。** 下書き承認後、Gmail で今すぐ送信するかドラフト保存するかを確認する。指示に従って実行する。

8. **結果を報告する。** 1段落：返信を送信/保存したか、返金を実行したかどうか。

## Approval gates

- **オーナーの明示的な確認なしに Stripe で返金を実行しない** — 常に金額・顧客名・メールアドレス・取引 ID を提示してから実行する。
- **オーナーのレビューなしに返信を送信しない。** 常に完全な下書きを先に提示する。
- **返信フロー中に顧客情報を新規登録しない。** 対応後に提案する。
- **複数の Stripe 取引が一致する場合は自動選択しない。** 全件提示してオーナーに選ばせる。
- **注文の詳細を推測しない。** Stripe に記録がない場合は下書き内にその旨を明記する。
