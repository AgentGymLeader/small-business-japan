# CRM Maintenance JP

## Quick start

kintone またはスプレッドシート（顧客管理）のデータを分析し、
重複・情報不足・活動停止の顧客・フォローアップ漏れを特定して整理計画を作成する。
自動での更新・削除は行わない。

```
オーナー: 「CRM整理して」
→ 顧客データの取得
→ 重複・情報不足・活動停止レコードの検出
→ フォローアップ漏れの特定
→ 整理計画レポートの作成
```

## Workflow

1. **対象ツールを確認する。** kintone、スプレッドシート（顧客管理）のどれを使っているか確認する。接続されていない場合はデータをエクスポートして貼り付けてもらう。

2. **重複を検出する。** メールアドレス・電話番号・会社名で重複している顧客レコードをリストアップする。確認のための一覧を提示する——自動マージはしない。

3. **情報不足のレコードを特定する。** 以下の必須フィールドが欠けているレコードを一覧化する：
   - 会社名 / 担当者名
   - メールアドレスまたは電話番号
   - 最終接触日

4. **活動停止の顧客を特定する。** 90日以上接触のない顧客を一覧化する。区分：
   - 成約済み顧客（定期フォローアップが必要かもしれない）
   - 商談中の見込み客（フォローアップが遅れている可能性）

5. **フォローアップ漏れを確認する。** 「フォローアップ予定日」が設定されていてかつ過去の日付になっているレコードをリストアップする。

6. **整理計画レポートを作成する。** 以下の形式でまとめる：

   ```
   CRM整理レポート — {日付}

   ✅ データ品質サマリー
   - 総レコード数: {件}
   - 重複候補: {件}
   - 情報不足: {件}
   - 90日以上接触なし: {件}
   - フォローアップ漏れ: {件}

   📋 推奨アクション
   1. 重複を確認してマージする（{件}件）
   2. 情報不足レコードを補完する（{件}件）
   3. 活動停止顧客に再接触する（{件}件）
   ```

   **個人情報保護法対応チェック**:
   - 開示・訂正・削除・利用停止の請求が来ていないかオーナーに確認する
   - 第三者提供している顧客データがある場合は同意の有無を確認する
   - 要配慮個人情報（健康情報等）が含まれていないかを確認する
   これらは「要確認」としてフラグするのみ。対応の実行はオーナーとシステム管理者が行う。

## Approval gates

- **顧客データの自動更新・削除・マージは行わない。** 一覧提示のみ。
- **個人情報（顧客の連絡先情報）を外部に送信しない。**
- **オーナーの明示的な承認なしにレコードを修正しない。**
- **個人情報保護法（APPI）に基づく開示・訂正・削除・利用停止の対応はオーナーとシステム管理者が行う。** AIが代わりに完了したとは言わない。
- **個人情報のコンプライアンス対応（同意取得・利用目的の確認）が完了したとは言わない。**
- **第三者提供・外部エンリッチメントをオーナーの確認なしに実行しない。**
